The flow

Everything rides on Shopify's native cart. Any product can enter. Quote-only products are flagged by tag/metafield — price hidden, checkout blocked. The merchant reviews + prices in native Drafts.

1. PDPQuote-only: price hidden, CTA → "Add to Quote". Native "Add to Cart" behavior.
2. Cart"Quote required" section appears above native cart. Checkout hidden if only quote items; mixed-cart intercept if both kinds.
3. Form/pages/request-quote renders the quote-only subset + contact form (name, email, project info, file upload).
4. DraftCustomer created + Draft Order with quote-request tag. Merchant edits pricing and clicks Send invoice in native Drafts admin.

UX / UI critical review

Prioritised punch list. Use the filters to slice by severity.

Stop
1. Cart line still shows price on quote-only items
+
Native cart row for a quote-only product still renders R 299,990.00 and contributes to the Estimated Total. This leaks MAP-protected pricing. Fix: per-line CSS hide via [data-qr-line-id] data attribute emitted by our JS after /cart.js hydration. Add a "Quote required" pill on the line instead.
Stop
2. /cart/<variant>:1?replace permalink bypass
+
Shopify's permalink adds to cart and force-redirects to /checkout, bypassing our cart intercept entirely. A quote-only variant entered this way can reach checkout at price. Fix: ship a Shopify Function (checkout.cart.validation.v1) that blocks any variant carrying the quote-only tag. Storefront JS alone can't close this.
Stop
3. Dev-store password still on; no bot challenge on submit
+
For real launch: disable storefront password. Add BotID or hCaptcha on /apps/quote/submit — honeypot + HMAC is enough for a known dev store but not for a public storefront.
High
4. "Add to Quote" button visually identical to Add to Cart
+
Same pill shape, same cart icon, same height. The semantic shift isn't felt. Swap the cart icon for a document icon (Heroicons document-text) or remove the icon on quote-only products. Consider a subtle outline variant.
High
5. No "Price on request" copy on quote-only PDPs
+
Right now the PDP just shows an empty gap where the price was. Luxury / MAP-restricted gear expects "Price on request" or "Contact for pricing" copy in that slot — it clarifies the flow, builds trust, and prevents the "is this broken?" reaction.
High
6. Submit is one click — no confirmation step
+
For a quote that the merchant will act on during business hours, a "Send request to <store>?" confirmation or inline review step would reduce accidental submits and double-submits. Low effort, high trust payoff.
High
7. File upload has no size/progress feedback
+
The 10 MB limit is enforced server-side; in the UI it's silent. Add a progress indicator, show selected filename, show size with green/red threshold based on the 10 MB cap.
Med
8. Request-quote page is visually plain
+
A dark theme to match the rest of the site, a subtle hero image of a home automation install, and a brand-consistent H1 treatment would lift perceived quality. Functional today; not yet luxurious.
Med
9. Admin settings page uses plain HTML (no Polaris)
+
Intentional — Polaris 13 has a React 19 peer-dep conflict. The page sticks out inside Shopify admin. A thin layer of Polaris-like spacing/typography (fonts, paddings, border radii) would make it feel native. Acceptable for v1; revisit when Polaris v14 ships with React 19 support.
Med
10. Mixed-cart: both CTAs are visually identical primary buttons
+
In mixed mode the native "Check out" and our "Request Quote" both render as primary black pills. Suggest: quote CTA takes the cream-amber accent of the "Quote required" frame; native Checkout stays brand-primary. Easier for the eye to separate.
Med
11. No empty-state CTA on /pages/request-quote
+
If the customer lands on the form with nothing in the cart, we show "Your quote is empty — browse the shop to add items." Add a "Browse collections" button directly there.
Low
12. Shopify Flow email workflows not installed
+
Set up the two workflows described in flows/README.md before launch. Without them customers get no confirmation email and the merchant has no notification except "new draft" in admin.
Low
13. storefront.js loads on every storefront page (~16KB)
+
Fine as-is but the script could short-circuit sooner when no work applies (About Us, blog pages, etc.) to save a tiny amount of main-thread time.
Low
14. Submit errors show raw backend codes
+
e.g. validation_failed bubbles up to the user. Translate to human copy in storefront.js.
A11y
15. Relabel doesn't update aria-label on the button
+
Button text reads "Add to Quote" visually but screen readers may still announce "Add to cart" if the theme set aria-label="Add to cart". Update aria-label in the relabel loop to match.
A11y
16. Form labels wrap inputs instead of using for/id
+
Accessible today via wrapping, but explicit <label for="…"> + <input id="…"> is more robust across assistive tech and theme-level label overrides.
A11y
17. Honeypot — belt and suspenders
+
Already aria-hidden="true" + tabindex="-1". Add autocomplete="off" + role="presentation" to ensure assistive tech fully skips it.
